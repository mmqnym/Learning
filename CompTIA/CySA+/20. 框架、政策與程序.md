# 框架、政策和程序

# 概述

- Enterprise Security Architectures（企業安全架構）：組織內安全架構的使用

- Prescriptive Frameworks（規範性框架）：
  - COBIT
  - 成熟度模型 (Maturity Models)
  - 風險導向框架 (Risk-based Frameworks)
    - NIST 網路安全框架 (CSF)

- Audits and Assessments（審計與評估）：網路安全中的審計和評估

- Continuous Monitoring（持續監控）：網路安全產業中的持續監控使用方法

# 企業安全架構（Enterprise Security Architecture）

- **Enterprise Security Architecture (ESA)**（企業安全架構）：定義用於保護企業免受各種風險的基準線、目標和方法的框架
  - 源自 IT 治理和 IT 服務管理概念
  - 提供政策清單、程序檢查表、活動指南和技術建議
  - 協助提供可外部驗證的法規遵循聲明

- **IT Service Management (ITSM)**（IT 服務管理）：專注於利害關係人選擇、部署和操作適當技術，為組織帶來最大效益的管理方式
  - 成功的 ITSM 需要基於框架的治理模式

- **Framework-based Governance**（基於框架的治理）：尋求降低與 IT 服務交付相關的風險
  - 提供可驗證的法規遵循方法
  - 特別適用於需要嚴格法規遵循的產業

- **法規遵循範例**：
  - PCI DSS（支付卡產業資料安全標準）：處理信用卡業務時必須遵循
  - HIPAA（健康保險流通與責任法案）：適用於醫療保健提供者
  - Sarbanes-Oxley（沙賓斯-奧克斯利法案）：影響金融公司或上市公司

- **常見企業安全架構框架**：
  - ITIL（IT 基礎架構資源庫）
  - COBIT（資訊與相關技術的控制目標）
  - TOGAF（開放群組架構框架）
  - ISO 20000（IT 服務管理國際標準）

- **專業發展考量**：
  - 學習知名框架如 ITIL 可提供更多職涯發展機會
  - 有助於晉升至管理層級

# 規範性框架

- **Prescriptive Framework**（規範性框架）：明確規定控制選擇與部署的框架，具有強制性
  - 通常由法規遵循需求所驅動
  - 例如醫療機構為符合HIPAA法規，可能採用規範性框架來協助達成

- **常見框架**：
  - ITIL
  - COBIT
  - ISO 27001（資訊安全框架）
  - PCI DSS（支付卡產業資料安全標準）
  - 框架選擇取決於組織需求和IT治理結構

- **Maturity Model**（成熟度模型）：企業安全架構框架的組件，用於評估安全性、控制選擇和使用的正規化與優化程度，並解決可能存在的差距
  - 導入框架後，幫助組織了解目前框架實施程度處於哪個階段
  - 目標是從被動反應（reactive）轉向主動預防（proactive）
  - 舉例來說：一企業欲導入 ISO27001，但沒有稽核壓力，此時就可以評估企業當前導入的成熟度，以及預期實現此框架到何種程度（X% 符合或完全符合）。

- **五階成熟度模型**：
  1. **Initial**（初始階段）：高度被動反應性質
  2. **Managed**（管理階段）：透過風險評估進行準備與緩解
  3. **Defined**（定義階段）：已定義政策和程序，但仍有部分時間處於被動狀態
  4. **Quantitative Management**（量化管理階段）：管理層全面監督風險，風險已記錄在風險登記冊中
  5. **Optimizing**（優化階段）：完全主動預防，使用風險驅動方法識別和緩解風險

- **成熟度模型應用**：
  - 協助組織根據預期目標進行自我審視
  - 幫助確定組織面臨的風險暴露程度
  - 並非所有組織都需要達到最高成熟度（第五級）
  - 成熟度目標應根據組織風險偏好和業務模型決定
  - 有些組織維持第三或第四級已足夠，可將風險控制在可接受水平

# 基於風險的框架

- **Risk-Based Framework**（基於風險的框架）：使用風險評估來優先選擇安全控制與投資的框架
  - 相較於規範性框架（Prescriptive Framework），更能適應不斷演變的威脅環境
  - 允許企業依據自身風險狀況制定獨特的安全措施

- **規範性框架的侷限性**：
  - 過於依賴檢查清單的方式
  - 可能導致實施對實際風險無效的措施
  - 例如：為了滿足框架要求而實施 SIEM，卻造成大量警報和誤報

- **NIST Cybersecurity Framework**（NIST 網路安全框架）：一種基於風險的框架，專注於 IT 安全而非 IT 服務提供
  - 涵蓋三個核心領域：框架核心、實施層級和框架配置文件

- **Framework Core**（框架核心）：識別五個網路安全功能
  - 識別（Identify）
  - 保護（Protect）
  - 檢測（Detect）
  - 回應（Respond）
  - 恢復（Recover）
  - 每個功能可進一步分為類別和子類別

- **Implementation Tiers**（實施層級）：評估核心功能如何整合到組織的整體風險管理流程中
  - 分為四個等級：
    - 部分（Partial）- 最低等級
    - 風險知情（Risk-informed）
    - 可重複（Repeatable）
    - 自適應（Adaptive）- 最高等級

- **Framework Profiles**（框架配置文件）：
  - 用於陳述目前網路安全成果和目標網路安全成果
  - 通過比較目前與目標配置文件，識別能有效縮小網路安全能力差距的投資
  - 建立當前安全狀態的基線，並制定改進計劃以達到目標狀態

- **法規遵循與基於風險的方法**：
  - 法規遵循通常屬於規範性方法（必須執行特定措施）
  - 對於沒有嚴格法規要求的組織，基於風險的框架更為適合
  - 實例：除了 PCI DSS 等特定法規外，許多組織可採用基於風險的方法來管理其安全性

# 產業框架

- **PCI DSS**（支付卡產業資料安全標準）：由主要信用卡公司創建的一套安全標準，用於保護敏感的支付卡資訊免受詐欺和資料外洩。
  - 適用於所有處理、儲存或傳輸支付卡資訊的組織
  - 包含六大控制目標：
    1. 建立和維護安全網路
    2. 保護持卡人資料
    3. 維護漏洞管理計劃
    4. 實施強大的存取控制措施
    5. 定期監控和測試網路
    6. 維護資訊安全政策
  - 需由合格安全評估師(QSA)進行稽核
  - 安全分析師的角色：協助分析安全系統、識別漏洞、實施控制措施、監控系統

- **CIS**（網際網路安全中心）：提供一套最佳實踐指南和安全控制的非營利組織，用於協助組織保護其 IT 系統。
  - CIS 控制分為三類：基本、基礎和組織
  - 基本控制是第一道防線，包括硬體和軟體清單與事件應變計劃
  - 基礎控制建立在基本控制之上，包括設備安全設定和安全更新
  - 組織控制是最後一層防禦，包括安全培訓計劃和風險評估
  - 安全分析師使用 CIS 控制評估組織安全狀態並識別需改進的領域

- **OWASP**（開放式網頁應用程式安全計畫）：旨在促進和改善網頁應用程式安全的非營利組織。
  - OWASP Top 10 是最嚴重的 10 大網頁應用程式安全風險清單
  - 每三年更新一次（2017、2021、2024 年）
  - 常見風險包括注入攻擊、認證與會話管理問題、跨站腳本、存取控制缺陷等
  - 安全分析師負責分析組織的網頁應用程式，確保不存在這些漏洞
  - 負責實施如輸入驗證、加密、存取控制等風險緩解措施

- **ISO 27000**（國際標準化組織 27000 系列）：提供管理資訊安全的框架。
  - ISO/IEC 27001 是主要標準，規定了資訊安全管理系統的要求
  - 規範三個主要領域：政策、程序和責任、組織結構和管理
  - 控制分為技術控制（加密、防火牆）和組織控制（存取控制、事件管理）
  - 需要詳細的文件記錄，作為稽核的合規證明
  - 安全分析師負責確保組織具有健全的事件管理流程

- **OSS TMM**（開源軟體測試成熟度模型）：用於評估和改進開源軟體品質及其測試流程的框架。
  - 定義了五個測試成熟度等級：初始、管理、定義、量化管理和優化
  - 初始：組織剛開始使用開源軟體，尚無測試流程
  - 管理：建立了基本測試流程，但尚未正式化
  - 定義：正式化的測試流程已與開發過程整合
  - 量化管理：一致地應用正式測試流程並使用指標衡量其有效性
  - 優化：組織根據指標和數據持續改進其開源軟體測試流程
  - 安全分析師協助組織達到更高的開源軟體測試成熟度

# 稽核與評估

## 主要概念

- **Audits and Assessments**（稽核與評估）：系統評估和稽核的流程，包含多種相關組件
  - 包括品質控制、品質保證、驗證與確認、評估與評價、稽核、定期審查和持續改進

## 詳細內容

- **Quality Control**（品質控制）：確定系統是否沒有缺陷或不足之處的流程
  - 例如製造汽車時，檢查產品是否符合品質標準
  - 發生在產品或系統製作期間的具體行動

- **Quality Assurance**（品質保證）：分析什麼構成品質以及如何測量和檢查品質的流程
  - 包含品質控制在內的整個品質管理計劃
  - 確保達到一定品質水準的完整程序

- **Verification and Validation (V&V)**（驗證與確認）：特別常見於軟體開發領域
  - **Verification**（驗證）：確保安全系統符合框架、法規環境要求或產品/系統達到設計目標的合規測試流程
    - 數位世界中的品質控制
    - 例如檢查程式碼是否執行其應有功能
  
  - **Validation**（確認）：確定安全系統是否適合用途，是否正確安裝和配置的流程
    - 關注系統是否適合用途（fit for purpose）
    - 在 ITIL 框架中，適合用途被稱為效用（utility）
    - 確認系統是否滿足特定用例的設計需求

- **Assessment and Evaluation**（評估與評價）：
  - **Assessment**（評估）：透過高度結構化的方式，針對絕對標準測試主體的流程
    - 基於檢查清單方法
    - 針對絕對標準進行比較
    - 例如：使用 83 點檢查清單認證的二手車
  
  - **Evaluation**（評價）：較不系統化的測試流程，旨在檢查測試主體的成果或實用性
    - 更多依賴評價者的判斷而非檢查清單或框架
    - 更可能使用比較測量
    - 更具流動性，更多基於意見

- **Audit**（稽核）：比評估或評價更嚴格的流程
  - 稽核人員根據預定基準比較組織，以識別需要改進的領域
  - 通常在受監管行業中需要進行（如支付卡產業、醫療數據處理等）
  - 針對已知標準進行嚴格比較
  - 通常是季度或年度進行

- **Scheduled Reviews**（定期審查）：
  - 類似於經驗教訓審查，但在固定時間間隔進行（如每季度或每年）
  - 審查內容包括：
    - 上一週期的主要事件
    - 可能影響公司網路安全狀態的威脅情報趨勢分析
    - 安全控制或系統的變更和添加，可能增加或減少風險
    - 採用或更新不同合規框架的進展

- **Continual Improvement**（持續改進）：
  - 透過識別缺陷和低效率，對產品和服務進行小幅增量改進的流程
  - 通過長期的小變更累積成大變更
  - 常見方法包括六標準差（Six Sigma）、戴明循環（Deming Cycle）、精益方法（Lean Method）和 ITIL 的七步改進方法
  - 一般流程包括：
    - 定義目標
    - 通過測量建立基準
    - 進行分析
    - 實施改進行動
    - 監控結果
    - 再次測量
    - 持續調整直到達成目標

# 持續監控筆記

- **Continuous Monitoring（持續監控）**：持續評估環境變化的技術
  - 能更快偵測新風險並改善業務運作
  - 是獲取管理組織風險所需重要資訊的持續努力

- **持續監控的好處**：
  - 提高情境感知能力（situational awareness）：了解哪些系統正在使用，哪些系統有問題
  - 能進行常規審計（routine audits）：不必等待季度或年度評估
  - 實時分析（realtime analysis）：發生問題時立即收到警報
  - 將組織從反應式（reactive）轉變為主動式（proactive）處理

- **持續監控常見問題**：
  - 過度依賴指標（metrics）而未妥善定義
  - 收集資料卻沒有人分析
  - 需建立能反映組織風險的可行動指標（actionable metrics）
  - 收集資料只為收集資料並非真正的持續監控

- **持續監控實施挑戰**：
  - 實施和維護複雜且耗時
  - 若需24/7全天候監控，需要人員輪班且成本高
  - 需依據風險偏好（risk appetite）決定監控時間範圍

- **CDM (Continuous Diagnostics and Mitigation)（持續診斷和緩解）**：
  - 美國國土安全部推出的計畫
  - 美國政府機構被要求採用持續安全監控
  - 提供政府機構識別網路安全風險的能力和工具
  - 基於潛在影響優先處理風險，使人員能先解決最重要問題

- **CDM 架構**：
  - 儀表板（dashboard）：彙總並以有用方式顯示資料
  - 資產管理（asset management）：回答「網路上有什麼」
  - 身份與存取管理（identity and access management）：回答「誰在網路上」
  - 網路安全管理（network security management）：回答「網路上發生什麼」
  - 資料保護管理（data protection management）：回答「資料如何被保護」
  - 各層級由不同系統和感測器組成，彙總資料後提供給管理者、分析師和主管
